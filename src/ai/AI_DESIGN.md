# SimpleAi — การออกแบบและการทำงาน

## ภาพรวม

`SimpleAi` เป็น AI สำหรับเกม Battle Spirits ที่ตัดสินใจในแต่ละ phase ของเทิร์น  
ไม่ได้ใช้ tree search หรือ scoring function — ใช้ heuristics ที่อ่านง่ายและแก้ไขได้ง่ายแทน

---

## โครงสร้างการตัดสินใจ

AI มีเมธอด 5 ตัว แต่ละตัวถูกเรียกจาก game engine ในช่วงเวลาต่างกัน:

```
เทิร์น AI:
  MAIN phase   → chooseDeploy()    เล่น Nexus ก่อน
               → chooseSummon()    เรียก Spirit
               → choosePlayMagic() เล่น Magic
  ATTACK phase → chooseAttackers() เลือกว่าจะส่งตัวไหนโจมตี

เทิร์นฝ่ายตรงข้าม (ระหว่างถูกโจมตี):
               → chooseBlock()     เลือกว่าจะบล็อคหรือไม่ และบล็อคด้วยตัวไหน
```

---

## ระบบจัดสรรและป้องกันคอร์ (AI Core Protection)

เพื่อป้องกันไม่ให้ AI ทำการร่ายการ์ดแล้วดึงคอร์จาก Spirit ที่อยู่บนบอร์ดมาใช้จนต่ำกว่า Level 1 (ซึ่งจะทำให้ Spirit บนบอร์ดดรอปมาที่ Level 0 และถูกทำลาย/Depleted) AI จึงมีระบบจัดการคอร์แบบอัตโนมัติภายในตัวเองก่อนตัดสินใจร่ายการ์ดใดๆ ใน Main Phase:

### 1. คำนวณคอร์ที่ใช้งานได้จริง (Usable Cores)
ก่อนที่ AI จะเลือกเล่นการ์ดเวทมนตร์ (`choosePlayMagic`), การ์ดเน็กซัส (`chooseDeploy`), หรืออัญเชิญการ์ดสปิริต (`chooseSummon`) AI จะตรวจสอบคอร์ที่สามารถดึงมาจ่ายได้จริงเท่านั้น โดยคำนวณจาก:
`คอร์ที่ใช้งานได้จริง (Usable Cores) = คอร์ใน Reserve + คอร์ทั้งหมดบน Nexus + คอร์ส่วนเกินบน Spirit (ที่เกินจากคอร์ขั้นต่ำในการรักษาสภาพ Level 1)`

สูตรการคำนวณคอร์บน Spirit แต่ละตัว:
`คอร์ส่วนเกินบน Spirit = สูงสุด(0, คอร์ทั้งหมดบน Spirit - คอร์ขั้นต่ำของ Level 1 ของการ์ดนั้นๆ)`

หากคอร์ที่จ่ายได้จริงมีค่าน้อยกว่าค่าร่ายและความต้องการขั้นต่ำ (เช่น Cost ของการ์ด + คอร์ขั้นต่ำสำหรับอัญเชิญสปิริตใหม่) AI จะข้ามการร่ายการ์ดใบนั้นทันที เพื่อไม่ให้บอร์ดตัวเองล่ม

### 2. ขยับคอร์ล่วงหน้าอย่างถูกกฎ (Reserve Preparation)
เมื่อ AI ตกลงใจจะเล่นการ์ดใบใดใบหนึ่งแล้ว ก่อนที่จะส่งดัชนีการ์ดกลับไปยัง Game Engine ตัว AI จะทำการดึงคอร์ส่วนเกินจาก Spirit และคอร์ทั้งหมดบน Nexus กลับมาพักไว้ที่ Reserve ของตนล่วงหน้าโดยใช้คำสั่ง **`game.moveCore(...)`** อย่างเป็นไปตามกฎของเกม:
- ดึงคอร์ปกติบน Spirit ที่ไม่จำเป็นสำหรับการรักษาสภาพ Level 1
- ดึง Soul Core บน Spirit หากมี Normal Core เหลือเพียงพอสำหรับสภาพ Level 1
- ดึงคอร์ทั้งหมดบน Nexus

การขยับคอร์ล่วงหน้านี้จะช่วยจัดสรรคอร์ให้อยู่ใน Reserve จนครบตามจำนวนที่ต้องการ ทำให้เมื่อ Game Engine เรียกการจ่าย Cost และการวางคอร์ ระบบจะดึงจาก Reserve ได้โดยตรงและสมบูรณ์ โดยไม่เกิดการดึงคอร์ผิดตัวจน Spirit บนบอร์ดถูกทำลาย

## แต่ละเมธอด

### `choosePlayMagic`

เล่น Magic การ์ดแรกที่เล่นได้ ไม่มีการเลือก  
ปัจจุบัน AI ยังไม่มีเหตุผลในการเลือกว่าจะเก็บ Magic ไว้หรือไม่

---

### `chooseDeploy`

Deploy Nexus แรกที่ deploy ได้ ไม่มีการเลือก

---

### `chooseSummon`

เรียก Spirit ที่ **BP สูงที่สุด** ที่สามารถเรียกได้ในมือ  
ถ้า BP เท่ากัน จะเลือกตัวที่ **cost ต่ำกว่า** (ประหยัด cores กว่า)

```
sort: BP สูง → cost ต่ำ → เลือกตัวแรก
```

---

### `chooseAttackers` — หัวใจหลัก

ตัดสินใจว่าจะส่ง Spirit ตัวไหนโจมตีในแต่ละเทิร์น  
ใช้ 4 โหมดตามลำดับความสำคัญ:

#### โหมด 1 — ตีปิด (Lethal Check)

**คำถาม**: ถ้าส่งตัวทั้งหมดโจมตี เราสามารถ reduce life ฝ่ายตรงข้ามเป็น 0 ได้แน่ๆ ไหม?

**วิธีคิด**:
- สมมติว่าฝ่ายตรงข้ามบล็อคตัวที่ BP สูงที่สุดของเราก่อน (worst case)
- ตัวที่เหลือ (N - B ตัว) จะผ่านเข้าโดยไม่ถูกบล็อค
- นับ symbols รวมของตัวที่ผ่านเข้า
- ถ้า guaranteed damage ≥ life ศัตรู → **ส่งทุกตัว**

```
เรียงตัวโจมตีตาม BP มากไปน้อย
ตัดตัวที่อาจถูกบล็อค (B ตัวแรก) ออก
ที่เหลือ = unblocked spirits
sum(symbols ของ unblocked) >= opp.life → LETHAL
```

> **ทำไมต้องเรียงตาม BP?**  
> ฝ่ายตรงข้ามมักจะบล็อคตัวที่ BP สูงเพื่อป้องกันตัวเองจาก BP comparison  
> ตัวที่ BP ต่ำกว่าจึงมีโอกาสสูงที่จะ "ลอดผ่าน" ไม่ถูกบล็อค

---

#### โหมด 2 — ตั้งรับ (Defensive Check, life ≤ 2)

**คำถาม**: ถ้าส่งตัวทั้งหมดโจมตีแล้วถูก exhaust จนไม่มีตัวบล็อค เราจะตายในเทิร์นถัดไปไหม?

**เงื่อนไข**: `player.life ≤ 2` และ `opp.symbols รวม ≥ our life`

**วิธีคิด**:
- เก็บตัวที่ BP สูงที่สุดไว้เป็น blocker (จำนวน = min(จำนวน opp spirits, ครึ่งหนึ่งของตัวเรา))
- ส่งตัวที่เหลือโจมตี
- ถ้าไม่มีตัวเหลือเลย → **ไม่โจมตีเลย**

```
เรียงตัวเราตาม BP มากไปน้อย
เก็บ keepCount ตัวแรกไว้บล็อค
ส่งที่เหลือโจมตี (ถ้ามี)
```

> **ทำไมเก็บตัว BP สูง?**  
> ตัว BP สูงมีโอกาสชนะ BP comparison ได้มากกว่า → บล็อคได้มีประสิทธิภาพกว่า

---

#### โหมด 3 — กดดันพร้อมมีเผื่อรับ (life = 3)

**เงื่อนไข**: `player.life = 3` และ `opp damage ≥ 3` และมีตัวมากกว่า 1 ตัว

**วิธีคิด**:
- เก็บตัวที่ BP สูงที่สุดไว้ 1 ตัวเป็น emergency blocker
- ส่งตัวที่เหลือโจมตีเพื่อสร้างแรงกดดัน

```
strongest = spirit ที่ BP สูงสุด → เก็บไว้
attackers = ที่เหลือทั้งหมด → โจมตี
```

---

#### โหมด 4 — กดดันเต็มที่ (Default)

เมื่อชีวิตยังดีอยู่ (> 3) หรือฝ่ายตรงข้ามไม่มีตัวคุกคาม  
**ส่งทุกตัวโจมตี** เพื่อสร้างแรงกดดันสูงสุด

---

### `chooseBlock`

เรียกทีละ attacker — ตัดสินใจว่าจะบล็อคด้วยตัวไหน (หรือไม่บล็อคเลย)

#### ลำดับการตัดสินใจ:

```
1. มี blocker ที่ BP >= attacker.BP ไหม?
   → บล็อคด้วยตัวที่ BP น้อยที่สุดที่ชนะได้ (เซฟตัวใหญ่ไว้)

2. ชนะ BP comparison ไม่ได้:
   a. defender.life <= attacker.symbols? → ตายแน่ถ้าไม่บล็อค → บล็อคด้วยตัวใหญ่สุด
   b. defender.life <= 2?               → วิกฤต → บล็อคด้วยตัวใหญ่สุด
   c. attacker.symbols >= 2 และ life <= 4? → symbols มาก เสียหายเยอะ → บล็อคด้วยตัวใหญ่สุด

3. ไม่มีเหตุผลพอ → ไม่บล็อค รับ damage ไป
```

#### เหตุผลที่เลือกบล็อคด้วย "ตัวเล็กที่สุดที่ชนะ":

การบล็อคด้วยตัวเล็กสุดที่ชนะ BP comparison หมายถึง:
- ทำลาย attacker ได้
- เซฟตัวที่ BP สูงกว่าไว้สำหรับโจมตีหรือบล็อคเทิร์นถัดไป

#### เหตุผลที่ block เมื่อ `symbols >= 2 และ life <= 4`:

ใน Battle Spirits, symbols = จำนวน life ที่หายไปต่อครั้ง  
ถ้า attacker มี symbols 2+ และเราไม่บล็อค เราเสีย 2 life ในครั้งเดียว  
เมื่อ life เหลือ 4 ลงมา การเสีย 2 life ต่อครั้งถือว่าอันตรายเกินพอที่จะยอมเสีย spirit ไปบล็อค

## ข้อจำกัดปัจจุบัน

| หัวข้อ | สถานะ |
|--------|--------|
| Magic timing | เล่นทันทีถ้าเล่นได้ ไม่รู้จักเก็บไว้ใช้ตอนวิกฤต |
| Summon priority | ดู BP อย่างเดียว ไม่คำนึง synergy กับการ์ดในมือ |
| Deploy timing | Deploy ทันทีที่ทำได้ ไม่ประเมิน board state |
| Lethal estimate | ใช้ worst-case (opp blocks best) ไม่คำนึงว่า opp อาจไม่มีตัวที่แข็งพอบล็อค |
| Multi-symbol spirits | ไม่ได้ prioritize การส่ง high-symbol spirits โจมตีก่อนเพื่อโอกาส lethal |
| Flash timing | ไม่มีการจัดการ Flash cards ในช่วง combat |

## ค่าที่ใช้ตัดสินใจ

| ค่า | แหล่งที่มา | ความหมาย |
|-----|-----------|----------|
| `spirit.symbols` | card data | จำนวน life ที่โจมตีได้ต่อครั้ง |
| `getEffectiveBP(spirit, card)` | engine function | BP จริงหลังคำนวณ level จาก cores + bpBoost |
| `player.life` | game state | ชีวิตที่เหลืออยู่ (เริ่มต้น 5) |
| `spirit.exhausted` | game state | true = โจมตีแล้วหรือถูก exhaust แล้ว |
